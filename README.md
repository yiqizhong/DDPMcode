# DDPMCODE

Product-UI page generation for Dell device-management screens, organized as a
**three-layer, category-isolated, slot-based** structure with **Devin skills** doing the
generation.

## Three layers

- **Design tokens** (`shared/tokens.css`) — shared visual constants; rarely change.
- **Category template** — framework **skills** at the repo-root `.agents/skills/`, namespaced
  `headset-*` (each holds a slot-based template asset), plus the category stylesheet
  `headset/headset.css`. Generation goes through the skills, not hand-written HTML.
- **Model data** (`headset/models/<MODEL>/`) — one folder per concrete model: its manifests
  (清单) plus the generated, content-baked pages.

Differences between products live in the model manifest, never in the template. **Cross-model**
variant and list content is generated by presence/absence — a not-selected product variant is never
CSS-hidden or pre-embedded. (The deliberate exception: **end-user interaction inside one device** —
segmented conditional panels, sub-function greying — may pre-embed + reveal via CSS `:has()`/`:checked`.)

## Skill location

Skills live at the repo-root `.agents/skills/` (where Devin discovers them — it does not scan
nested dirs) and are namespaced `headset-…` to keep category isolation. This reconciles
methodology §9.5 (which nests skills per category) with Devin's root-only skill discovery —
the isolation intent is preserved via naming; only the folder location differs. See
`AGENTS.md`.

## Pilot

`headset/` is the current pilot category — framework skills (`headset-gen-homepage`,
`headset-gen-subpage`, `headset-function`), layout, and category rules. It now has a few **real,
usable cards** in `templates/functions/` (`eq-audio`, `promotion-download`, `single-control`) plus
**reference/demo cards** in `templates/examples/` (`collaboration`, `auto-power-off`, `noise-control`
— NOT id-routed), plus two committed **test fixture** models (`FIXTURE`, `HS-DEMO`) under
`headset/models/` whose manifests are tracked while their generated `.html`/`READ-LOG` are gitignored.
There are **no dedicated `headset-function-<id>` skills** — known functions default to `functions/<id>.html`
snapshots that grow from real manifests (methodology §9.4). Other device categories get their own
isolated folder later, sharing only `shared/tokens.css`.

## Layout

```
.agents/skills/                       # repo ROOT — Devin discovers skills here (folders with SKILL.md)
  headset-gen-homepage/   SKILL.md + templates/home-frame.html      # home-page frame (copy + fill)
  headset-gen-subpage/    SKILL.md + templates/subpage-frame.html   # sub-page frame (holds any sub-page)
    templates/functions/              # id-routed registry — REAL cards (eq-audio, promotion-download, single-control)
    templates/examples/               # reference/demo cards — NOT id-routed (collaboration, auto-power-off, noise-control)
  headset-function/       SKILL.md + templates/function-frame.html  # no-snapshot card builder (shell + atoms)
  headset-shared/                     # NOT a skill — snippets shared by the gen skills:
    components/  connection/  icons/  feature-button.html
shared/tokens.css                     # design-token layer
headset/
  AGENTS.md                           # category map: routing + non-negotiables
  headset.css                         # category layout (uses tokens via var())
  models/FIXTURE/ models/HS-DEMO/     # committed test fixtures: manifests tracked, generated .html/READ-LOG gitignored
docs/                                 # navigation.md (map) · component-catalog.md · function-card-architecture.md (journal) · methodology.md (reference)
AGENTS.md                             # cross-category map
README.md                             # this file
```

**Start here:** [`docs/navigation.md`](docs/navigation.md) — what to read/change for a given task.
Then `AGENTS.md` (cross-category map), `headset/AGENTS.md` (category routing), and
[`docs/component-catalog.md`](docs/component-catalog.md) (the reusable components).
