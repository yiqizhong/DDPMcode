From “Product Requirement” to “HTML pages” — the generation pipeline

How a headset companion-app UI is generated automatically from a single product requirement.

Core idea: the manifest is the single source of truth; every step except ① is a deterministic script + mechanical gate; any machine-detectable error self-corrects and re-runs.

🛠 The system’s 5 skills

Every step in the pipeline is carried by these skills — the system’s toolbox.

headset-gen-subpageThe engine. The subpage framework plus all validate / render / verify / coverage / regen scripts live here.
headset-gen-homepageThe home page’s (device identity · battery · connection · feature entries) schema contract + frame templates; it contains only SKILL.md + templates — the home page’s validation (validate-home.py) and rendering (render-home.py) actually live in the headset-gen-subpage engine.
headset-functionThe function-card shell (function-frame); assembles unknown functions on the spot and, once reused, settles them into a snapshot.
headset-sharedReusable component snippets (toggle / slider / segmented / dropdown / preset-grid…), icons, connection blocks, CSS.
shared-gen-walkthroughSchema validation and rendering for the onboarding walkthrough.

manifest = single source of truth　·　output is fully regenerable from the manifest, hand-edits forbidden　·　a human is needed only when intent must be guessed
